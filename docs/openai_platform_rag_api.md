# OpenAI Platform RAG API (P29)

## Purpose
Server-side guarded RAG gateway over selected ScienceR-Dsim artifacts.

## Architecture
Manifest builder -> policy check -> dry-run sync plan -> query client -> optional API server.

## Why Responses API + file_search
Supports retrieval over uploaded vector-store artifacts with explicit include fields.

## Vector store strategy
Default is mock/dry-run. Live uploads require `--mode live --live --confirm-upload` and key.

## Artifact manifest
Generated by `python -m tools.openai_rag.artifact_manifest`.

## What gets indexed
Selected ToL safe outputs, empirical report artifacts, local ops outputs, and selected Obsidian notes.

## What is excluded
`.env`, secrets, `data/*`, raw EEG binaries, and private credential patterns.

## API key handling
Server-side env-only key, never emitted in logs or outputs.

## Mock/dry-run/live modes
Mock and dry-run are offline; live is explicitly opt-in.

## API endpoints
`/health`, `/artifacts/manifest`, `/rag/status`, `/rag/sync-plan`, `/chat`, `/query`.

## Chatbot usage
Use `/query` in mock mode for deterministic local summary without network calls.

## ChatGPT Action / MCP future path
OpenAPI schema included for future external action binding.

## Guardrails
Unsafe ontology/clinical certainty claims are blocked/reframed to candidate telemetry language.

## Cost controls
No uploads/calls by default; dry-run sync shows what would happen.

## How to run
See Make targets: `openai-rag-policy-check`, `openai-rag-manifest`, `openai-rag-dry-run-sync`, `openai-rag-query-mock`, `openai-rag-api-smoke`.
